# PROJECT_CONTEXT.md

## Project
WoodOps.Workers

## Purpose
WoodOps.Workers hosts background execution for the WoodOps platform.

This project runs asynchronous and scheduled processing tasks that operate on the WoodOps platform model and processing pipeline.

Workers enable continuous platform activity without requiring direct user interaction.

Examples include ingestion pipelines, polling integrations, scheduled processing, and long-running computations.

---

## Architectural Role
WoodOps.Workers functions as the **background execution host** of the platform.

Responsibilities include:

• executing background processors  
• running ingestion pipelines  
• polling external systems  
• scheduling recurring tasks  
• reacting to events generated by the platform  

Workers execute logic provided by the platform kernel and integrations but should not define the platform’s model or core capabilities.

This project is typically deployed as a **separate Docker container**.

---

## Owns

### Background Processing Host
Provides the runtime environment for executing background tasks.

Examples:

- processor execution loops
- scheduled jobs
- long-running operations
- batch processing

---

### Ingestion Pipelines
Runs ingestion and synchronization processes that retrieve information from external systems.

Examples:

- polling external APIs
- consuming file drops
- scheduled data imports
- webhook processing

These pipelines typically rely on code from WoodOps.Integrations and WoodOps.Core.

---

### Event-Driven Processing
Handles asynchronous tasks triggered by events in the system.

Examples:

- processing newly ingested data
- recalculating projections
- triggering downstream processing pipelines

---

### Operational Automation
Supports automated operational tasks.

Examples:

- system maintenance jobs
- background analysis
- automated data enrichment

---

## Does Not Own

WoodOps.Workers must **never contain**:

• platform model definitions  
• integration translation logic  
• core processing primitives  
• UI components  
• API controllers  
• LLM orchestration logic  

These responsibilities belong to:

• WoodOps.Core  
• WoodOps.Integrations  
• WoodOps.Intelligence  
• WoodOps.Host  
• WoodOps.Portal  

Workers execute logic but should not define platform architecture.

---

## Inputs

WoodOps.Workers receives:

• work requests from the platform kernel  
• scheduled job definitions  
• ingestion instructions  
• configuration supplied by the runtime environment  

Inputs may originate from:

• the host service  
• integration triggers  
• scheduled execution

---

## Outputs

WoodOps.Workers produces:

• processed model objects  
• new or updated projections  
• background task results  
• operational events and diagnostics

Results are typically written through the platform persistence layer.

---

## Allowed Dependencies

WoodOps.Workers may depend on:

• WoodOps.Core  
• WoodOps.Integrations  
• WoodOps.Intelligence  

These dependencies provide the logic that workers execute.

---

## Dependency Restrictions

WoodOps.Workers must **not be referenced by**:

• WoodOps.Core  
• WoodOps.Integrations  
• WoodOps.Intelligence  

Workers exist at the runtime edge of the system.

Kernel libraries must remain independent of the execution host.

---

## Internal Namespace Guidance

Typical namespace layout:

WoodOps.Workers.Jobs  
WoodOps.Workers.Pipelines  
WoodOps.Workers.Schedulers  
WoodOps.Workers.Host  

Jobs should be thin execution wrappers that call processing primitives from the platform kernel.

---

## Change Guidance

Changes should focus on:

• improving background processing reliability  
• improving scheduling and orchestration  
• improving ingestion throughput  
• improving operational observability  

Avoid embedding platform logic inside workers.

If platform logic begins appearing here, it likely belongs in:

• WoodOps.Core  
• WoodOps.Integrations  
• WoodOps.Intelligence

---

## Notes

WoodOps.Workers represents the **asynchronous execution engine** of the WoodOps platform.

It allows the platform to continuously process data, synchronize with external systems, and run automated operations independent of user interaction.
